    ```typescript
    export interface ScoutOptions {
      projectRoot: string;

      name: string;
      path: string;
      loc: number;
      languages: string[];
      frameworks: string[];
      entryPoints: string[];
    }

    export interface CiCdInfo {
      tool: string;           // "github-actions", "gitlab-ci", "jenkins", "circleci", etc.

    }

    export interface ScoutResult {
      projectType: string;

      cicd: CiCdInfo[];
      durationMs: number;
      manifestPath: string;    // path to written service-manifest.md
    }
    ```

    **Function: `runScout(options: ScoutOptions): Promise<ScoutResult>`**

    Implementation:
    1. Record start time.
    2. Try to read existing config.yml via `loadConfig(options.projectRoot)`. If available, extract project info (name, type, languages, services). If not, call `detectProject(options.projectRoot)` directly (per D-04: extend onboard's detectProject output).
    3. Enrich with targeted file reads:

    **LOC counting:** For each service path (or root for single project):
    - Walk the directory recursively (same ignore pattern as builder: skip node_modules, dist, build, .git, vendor, etc.)
    - Count lines of all source files (`.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.java`, `.go`, `.rs`, `.rb`, `.php`, `.cs`, `.swift`, `.kt` -- broader than parser support, since LOC is language-agnostic)
    - Sum to get approximate LOC

    **Framework detection:** Read package.json (root and per-service) dependencies:
    ```typescript

      "fastapi": "FastAPI",
      "svelte": "Svelte",
      "tailwindcss": "Tailwind CSS",
      "prisma": "Prisma",
      "@prisma/client": "Prisma",
    };
    ```
    Check both dependencies and devDependencies keys against these patterns.

    **Entry point detection:**
    - Read package.json `main`, `module`, `exports` fields
    - Check for common entry files: `src/index.ts`, `src/index.js`, `src/main.ts`, `src/app.ts`, `src/server.ts`, `app.py`, `main.py`, `manage.py`
    - Return list of detected entry points

    **CI/CD detection:**
    - Check for: `.github/workflows/` (GitHub Actions), `.gitlab-ci.yml` (GitLab CI), `Jenkinsfile` (Jenkins), `.circleci/config.yml` (CircleCI), `.travis.yml` (Travis), `azure-pipelines.yml` (Azure DevOps), `bitbucket-pipelines.yml` (Bitbucket), `Dockerfile` (Docker)
    - Return tool name and config file path

    4. Build ServiceEntry array from enriched data.

    5. Generate `service-manifest.md` following UI-SPEC copywriting contract exactly:

    ```markdown
    ---
    generated: "{ISO 8601 timestamp}"
    generator: "scout"
    phase: 2
    scout_duration_ms: {durationMs}
    project_type: "{single|monorepo|polyrepo}"
    ---

    # Service Manifest

    ## Services

    | Service | Path | LOC (approx) | Languages | Frameworks | Entry Points |
    |---------|------|--------------|-----------|------------|--------------|
    | {name} | `{path}` | {loc} | {languages joined} | {frameworks joined} | `{entryPoints joined}` |
    ...

    ## CI/CD

    | Tool | Config File |
    |------|-------------|
    | {tool} | `{configFile}` |
    ...
    ```

    BUT still include the root project as the single row in a simplified table showing root-level info.
