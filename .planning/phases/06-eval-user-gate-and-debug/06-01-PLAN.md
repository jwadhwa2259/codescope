    Create `src/eval/eval-agent.ts` following the agent module pattern (Phase 2 D-05: Options + Result + async function):

    1. `tokenEstimate(text: string): number` -- Returns `Math.ceil(text.length / 4)` as rough token approximation per RESEARCH.md Pitfall 1. Exported for testing.

    2. `chunkVerifyResult(options: EvalOptions, threshold: number = 50_000): EvalOptions[]` -- Splits large verify results into chunks per D-22:
       - Serialize verifyResult to JSON string, call tokenEstimate on it
       - If under threshold: return `[options]` (single chunk, no splitting)
       - If over threshold: group by file context. Convention violations, blast radius entries, and code review findings each reference files. Group these by file path into clusters.
       - Build one EvalOptions per cluster where each cluster's verifyResult contains only the violations/findings for that cluster's files, plus runtime results (shared across all chunks since they're not file-specific)
       - Each chunk retains the full scopeContractPath, planPath, coordinationPath, researchPath (per D-22: scope_compliance and completeness get full scope contract regardless of chunk)
       - If a single file cluster still exceeds threshold, include it as-is (LLM will handle truncation gracefully)
       - Return array of EvalOptions chunks

    3. `deduplicateFindings(findings: EvalFinding[]): EvalFinding[]` -- Deduplicates findings across chunks:
       - Group by finding.id
       - When duplicates exist, keep the one with highest severity (ERROR > WARN > INFO)
       - Return deduplicated array preserving original order of first occurrence

    4. `buildEvalPrompt(options: EvalOptions): string` -- Assembles eval prompt with 6 sections:
       - H1 header: "Eval Agent: LLM-as-Judge" with role description
       - Enabled criteria list (filtered by enabledCriteria booleans, skip disabled)
       - Scope contract by file reference: `Read: \`${options.scopeContractPath}\``
       - Execution plan by file reference: `Read: \`${options.planPath}\``
       - Coordination log by file reference: `Read: \`${options.coordinationPath}\``
       - Verify results as inline JSON: `JSON.stringify(options.verifyResult, null, 2)` in a code block (per D-01)
       - Research by reference (only if researchPath is not null): `Read: \`${options.researchPath}\``
       - Git diff instruction: `Run: \`git diff HEAD\` to see the actual changes`
       - Output format instruction: JSON array schema matching EvalFinding type, plus per-criterion PASS/FAIL statement
       - Sections joined with `\n\n`

    5. `parseEvalFindings(rawResponse: string): EvalFinding[]` -- Extracts findings JSON from LLM response:
       - Try direct JSON.parse first
       - If fails, look for ```json ... ``` code block and parse contents
       - If both fail, return empty array (garbled response)
       - Validate each finding has required fields (criterion, file, line, description, severity)
       - Generate `id` for each finding as: `eval-${criterion}-${file.replace(/[^a-z0-9]/gi, '-')}-${Math.floor(line / 5) * 5}` (5-line bucket per Research open question 2)
       - Discard findings with invalid criterion or severity values

    6. `groupFindingsByCriterion(findings: EvalFinding[]): Map<EvalCriterion, EvalFinding[]>` -- Groups by criterion, sorts within each group: ERROR first, then WARN, then INFO (per D-07)

    7. `runEval(options: EvalOptions, callbacks: EvalCallbacks): Promise<EvalResult>` -- Core eval logic:
       - Record start time
       - Call `callbacks.onProgress("## Evaluating changes...")`
       - Chunk if needed: `const chunks = chunkVerifyResult(options)` per D-22
       - If chunks.length > 1: `callbacks.onProgress(\`Chunking eval into ${chunks.length} parts (~50K token threshold)...\`)`
       - For each chunk, build prompt via buildEvalPrompt and dispatch via dispatchEvalAgent with retry (see below)
       - Merge all findings from all chunks, then deduplicate via deduplicateFindings
       - **Retry logic per D-26:** Wrap each `callbacks.dispatchEvalAgent(prompt)` call in a try/catch:
         - On first failure: log `callbacks.onProgress("Eval dispatch failed, retrying once...")` and retry the same call
         - On second failure: do NOT throw. Instead, mark all enabled criteria as `{ status: "PASS", findings: [], detail: "Unavailable: {error.message}. Verify results still valid." }`. Set overallStatus to "PASS" (per D-26: pipeline continues to summary). Return early with this result.
       - Filter parsed findings against options.ignorePatterns (match on pattern substring in description, scope glob match on file, criterion exact match)
       - Score each enabled criterion: criterion has findings with severity ERROR -> FAIL, WARN-only -> PASS, no findings -> PASS
       - Mark disabled criteria as SKIPPED with detail "Disabled in config"
       - Set overallStatus: any FAIL criterion -> FAIL, else PASS
       - Return EvalResult with timing_ms
