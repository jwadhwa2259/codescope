# Phase 1 — UI Design Contract

> Visual and interaction contract for the CodeScope plugin's text-based interfaces. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — CLI plugin, no browser UI |
| Icon library | none — no graphical rendering |
| Font | terminal default (user's terminal font) |

**Context:** CodeScope is a Claude Code plugin. All user-facing "UI" is text-based: Claude Code skill prompts, AskUserQuestion menus, MCP tool JSON responses, terminal log output, and generated Markdown/YAML files. There are no browser-rendered components, no CSS, no visual design tokens.

---

## Spacing Scale

Not applicable. This phase has no pixel-rendered UI. All output is terminal text or structured data (JSON, YAML, Markdown).

Exceptions: none

---

## Typography

Not applicable. All text renders in the user's terminal font at the terminal's configured size. No font size, weight, or line-height control.

---

## Color

Not applicable. No color tokens are specified. Terminal output uses plain text only — no ANSI color codes in MCP responses or generated files.

---

## Interaction Contract

This section replaces the standard visual design contract. Phase 1's interaction surfaces are:

### Surface 1: Onboarding Skill (`/codescope:onboard`)

The onboarding skill runs as a Claude Code skill in the main session context. It uses AskUserQuestion menus for structured input.

**Flow (3 sequential phases per D-05):**

| Step | Action | Input Method | Fallback |
|------|--------|-------------|----------|
| 0 | Prerequisite check (Node.js 22+, WASM grammars) | Automatic | Block with fix instructions (D-07) |
| 0b | Existing config check | Automatic | Offer "Update existing config" or "Start fresh" (D-09) |
| 1 | Project detection (type, languages, build/test/E2E) | Auto-detect, present for confirmation (D-02) | Manual entry via menus (D-04) |
| 1b | Returning user check | Read ~/.codescope/global-memory.md | Offer "Use same setup" vs "Customize" (D-03) |
| 2 | Agent model selection (6 agents) | Show defaults, accept all or override (D-06) | No per-agent walkthrough |
| 3 | Workflow preferences (4 areas) | Structured menus | Thorough defaults applied (D-12) |
| 4 | Write config.yml + directory tree | Automatic | Show brief summary + CTA for bootstrap (D-08) |

### Surface 2: MCP Tool Responses

All 11 MCP tools return JSON via StdioServerTransport. Responses are consumed by Claude Code (not rendered to users directly).

**Response format contract:**

| State | Schema | Example |
|-------|--------|---------|
| Success | `{ content: [{ type: "text", text: JSON.stringify(result) }] }` | Tool-specific structured data |
| Not bootstrapped | `{ content: [{ type: "text", text: JSON.stringify({ status: "not_bootstrapped", message: "Run /codescope:bootstrap first", tool: "<tool_name>" }) }] }` | Per D-46 |
| Validation error | `{ isError: true, content: [{ type: "text", text: "..." }] }` | Zod validation failure message |

### Surface 3: Stub Skill Responses

Unimplemented skills (bootstrap, orient, settings, review-learnings) display a single message per D-43:

> This skill will be available after Phase N.

### Surface 4: Generated Files

| File | Format | Consumer |
|------|--------|----------|
| config.yml | YAML with schema_version: 1 | Machine-parsed by CodeScope tools |
| .claude/codescope/.gitignore | gitignore syntax | Git |
| ~/.codescope/global-memory.md | Markdown | Claude Code agents |

## Config Schema Contract

The config.yml structure serves as a data contract between the onboarding skill (writer) and all downstream tools (readers).

**Top-level keys (per D-14, D-10):**

```yaml
schema_version: 1
project:
  name: string
  type: single | monorepo | polyrepo
  languages: string[]
  root: string
  services: # optional, monorepo only
    - name: string
      path: string
      build: string?
      test: string?
  build_command: string?
  test_command: string?
  e2e_tool: string?
  e2e_command: string?
agents:
  researcher: { model: haiku | sonnet | opus | inherited }
  convention_detector: { model: haiku | sonnet | opus | inherited }
  risk_analyzer: { model: haiku | sonnet | opus | inherited }
  learning_synthesizer: { model: haiku | sonnet | opus | inherited }
  eval_judge: { model: haiku | sonnet | opus | inherited }
  debug: { model: haiku | sonnet | opus | inherited }
orient:
  verbosity: brief | detailed
  clarification: thorough | minimal | auto
  research_sources: string[]
  max_research_time: number
eval:
  mode: interactive | auto-debug | auto-skip-minor
conventions:
  detection_threshold: number  # default 80
  min_files: number            # default 10
  strictness: suggest-only | warn | block
```

**Validation:** Zod schema at load time (D-15). Zod errors surface as the error copy defined above.

## MCP Tool Response Schema Contract

All MCP tool responses follow the MCP SDK content format. The internal JSON structure for `codescope_status` (the only functional tool in Phase 1):

```typescript
interface StatusResponse {
  config_exists: boolean;
  bootstrap_completed: boolean;
  last_bootstrap: string | null;    // ISO 8601 date
  graph_nodes: number;
  graph_edges: number;
  dependency_health: {
    node_version: string;
    node_compatible: boolean;
    sqlite_available: boolean;
    wasm_grammars_available: boolean;
    ast_grep_available: boolean;
  };
  plugin_version: string;
}
```

Stub tools (10 remaining) return:

```typescript
interface StubResponse {
  status: "not_bootstrapped";
  message: "Run /codescope:bootstrap first";
  tool: string;  // tool name
}
```
