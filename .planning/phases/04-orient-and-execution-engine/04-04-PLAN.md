    Export `ResearchOptions { projectRoot: string; taskSlug: string; task: string; analysisResult: AnalysisResult; scopeContract: ScopeContract; outputDir: string }`:

    Export `extractResearchTopics(projectRoot: string, affectedFiles: AffectedFile[]): ResearchTopic[]`:
    - Load graph via getGraph(projectRoot)
    - For each affected file, look at graph node's outgoing IMPORTS edges to find imported libraries/packages
    - Identify external libraries (those resolving to node_modules) vs internal modules
    - For each external library, compute impactScore = `centrality of importing file * number of files that import this library` (use graph traversal)
    - Deduplicate by library name, take max impactScore
    - Sort descending by impactScore
    - Mark as 'skipped' with reason `'low graph impact (score {N}, {M} files)'` if impactScore < 0.1

    Export `rankTopics(topics: ResearchTopic[], config: { researchSources: string[]; maxResearchTime: number }): ResearchTopic[]`:
    - Per D-11: high-impact topics (score >= 0.5) get source 'context7' + 'web_search'
    - Mid-impact (0.1 <= score < 0.5) get source 'context7' only
    - Low-impact (score < 0.1) get source 'skipped'
    - If 'context7' not in config.researchSources, upgrade mid-impact to 'web_search'
    - If 'web_search' not in config.researchSources, downgrade high-impact to 'context7' only
    - Return re-annotated topics

    Export `buildResearchPrompt(task: string, topics: ResearchTopic[], scopeContract: ScopeContract, config: { maxResearchTime: number }): string`:
    - Build a sub-agent prompt string (this is what gets passed to the Task/Agent tool)
    - Include:
      - Task description and scope contract summary (In Scope items)
      - For each non-skipped topic: library name, what to research, which source to use
      - Time budget: `Complete research within ${config.maxResearchTime} seconds`
      - Output format instructions: write to research.md with sections matching UI-SPEC format
      - Per D-13: note which research findings are most relevant to which plan aspects
    - Keep prompt under ~5K tokens (estimate by character count / 4)

    Export `parseResearchOutput(content: string): ResearchOutput`:
    - Parse a research.md file content into structured ResearchOutput
    - Extract: completedAt from `**Completed:**`, topicsResearched from `**Topics researched:**`, topicsSkipped from `**Topics skipped:**`
    - Parse sections: Relevant APIs, Best Practices, Known Issues, Version-Specific Notes, Skipped Topics

    Export `writeResearchArtifact(output: ResearchOutput, outputDir: string): string`:
    - Write research.md to outputDir matching the UI-SPEC format (lines 208-238)
    - Return the file path

    Export `runResearch(options: ResearchOptions): Promise<ResearchOutput>`:
    - Extract topics from analysis result
    - Load config for research_sources and max_research_time
    - Rank topics
    - Build sub-agent prompt
    - NOTE: The actual sub-agent spawning happens in the pipeline orchestrator (Plan 06), not here. This function prepares the prompt and processes the output.
    - If no non-skipped topics: write a minimal research.md noting "All affected libraries are internal -- skipping external research" and return
    - Return ResearchOutput with the prompt for the orchestrator to dispatch

    **src/orient/planner.ts**: Implement execution plan generation per ORNT-09, ORNT-10, D-14.

    Export `PlannerOptions { projectRoot: string; taskSlug: string; task: string; scopeContract: ScopeContract; analysisResult: AnalysisResult; researchOutput: ResearchOutput | null }`:

    Export `buildPlannerPrompt(options: PlannerOptions): string`:
    - Build a sub-agent prompt for the planner agent that will generate the execution plan
    - Include in prompt:
      - Task description and scope contract (In Scope / Out of Scope items)
      - Analysis results: affected files with risk levels, blast radius summary, conventions in scope, cross-community impact
      - Research highlights: key APIs, best practices, known pitfalls (if research was done)
      - Available conventions and golden files for affected areas
      - Instruction to produce agent assignments with:
        - Agent name (descriptive, like "auth-api-agent", "frontend-ui-agent")
        - Task description for each agent
        - Exclusive write files (files only this agent modifies)
        - Read-only files (files this agent references)
        - Dependencies (which other agents must complete first)
        - Estimated tokens and timeout
      - Instruction to use hybrid strategy per D-31/D-32:
        - Independent agents (no shared files, no dependencies) -> parallel
        - Dependent agents (blockedBy chains) -> sequential ordering
        - Max concurrent agents from config.execute.max_agents_concurrent
    - Keep prompt under ~8K tokens

    Export `parsePlanOutput(content: string): ExecutionPlan`:
    - Parse a plan markdown (as returned by the planner sub-agent) into ExecutionPlan
    - Extract agent blocks: look for `### Agent: {name}` sections
    - Parse fields: Wave, Task, Files (exclusive write), Files (read-only), Conventions, Golden files, Depends on, Estimated tokens, Timeout
    - Extract execution order table
    - Build agents array and waves from parsed data
    - Set status to 'PENDING', createdAt to current ISO timestamp
    - Compute strategy from wave structure: 'sequential'/'parallel'/'wave-based'

    Export `writePlanArtifact(plan: ExecutionPlan, outputDir: string): string`:
    - Write plan to `plans/[task-slug].md` in the UI-SPEC format (lines 240-283):
      - H1 with task-slug
      - Metadata: Created, Status, Strategy, Estimated agents, Estimated total tokens
      - H2 "Agents" with H3 per agent showing all fields
      - H2 "Execution Order" table: Wave, Agents, Mode, Files
      - H2 "Validation" with checkbox list of validation results
      - H2 "Removed by User" (empty by default, per D-17)
    - Return the file path

    Export `runPlanner(options: PlannerOptions): Promise<{ prompt: string; plan: ExecutionPlan | null }>`:
    - Build the planner prompt
    - NOTE: Actual sub-agent dispatch happens in pipeline (Plan 06). This returns the prompt.
    - If called with a pre-built plan (for testing), validate and return it
    - Return the prompt string for the orchestrator to dispatch

    **src/orient/validation.ts**: Implement plan validation with auto-fix per D-19 through D-23.

    Export `validatePlan(plan: ExecutionPlan, scopeContract: ScopeContract): ValidationResult`:
    - Run three validation checks using wave-scheduler functions:
      1. `validateFileOverlap(plan.agents, plan.waves)` -- EXEC-10
      2. `validateDependencyOrdering(plan.agents, plan.waves)` -- D-19(2)
      3. `validateScopeCoverage(plan.agents, scopeContract.inScope)` -- D-19(3)
    - Combine all checks
    - Return `{ passed: allChecks.every(c => c.status === 'PASS' || c.status === 'AUTO-FIXED'), checks: allChecks, autoFixAttempts: 0 }`

    Export `autoFixPlan(plan: ExecutionPlan, validationResult: ValidationResult, maxAttempts: number = 2): { plan: ExecutionPlan; result: ValidationResult }`:
    - Per D-20: mechanical errors auto-fixed, structural errors escalated
    - **Mechanical errors** (auto-fixable):
      - File overlap (FAIL on file-overlap check): move the second agent of the conflicting pair to the next wave, recalculate wave schedule via `buildWaveSchedule()`
      - Dependency ordering (FAIL on dep-ordering): reassign wave numbers via `buildWaveSchedule()` which handles topological sort
    - **Structural errors** (escalate as WARNING):
      - Scope coverage gaps: mark as WARNING with detail listing uncovered items
    - Loop up to maxAttempts:
      - Apply mechanical fix
      - Re-validate
      - If passed, return updated plan with AUTO-FIXED status on fixed checks
    - If still failing after maxAttempts: return plan with unfixed checks as WARNING + detail "Could not auto-fix: {reason}"

    Export `writeValidationSection(checks: ValidationCheck[]): string`:
    - Format validation checks as the "## Validation" section matching UI-SPEC format:
      - `- [x] {name}: **PASS**`
      - `- [x] {name}: **AUTO-FIXED** -- {detail}`
      - `- [ ] {name}: **WARNING** -- {detail}`
      - `- [ ] {name}: **FAIL** -- {detail}`
