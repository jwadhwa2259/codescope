       ```typescript
       function detectDefaultBranch(projectRoot: string): string {
         try {
           const ref = execFileSync("git", ["symbolic-ref", "refs/remotes/origin/HEAD", "--short"], {
             cwd: projectRoot, encoding: "utf-8", stdio: ["pipe", "pipe", "pipe"],
           }).trim();
           // ref is like "origin/main" -- strip "origin/" prefix
           return ref.replace(/^origin\//, "");
         } catch {
           // Fallback: try common names
           try {
             execFileSync("git", ["rev-parse", "--verify", "main"], {
               cwd: projectRoot, encoding: "utf-8", stdio: ["pipe", "pipe", "pipe"],
             });
             return "main";
           } catch {
             return "master";
           }
         }
       }
       ```

    Steps:
    1. Guard: `isBootstrapped(projectRoot)` -> errorResponse "NOT_BOOTSTRAPPED"
    2. Resolve diff: call `resolveDiff(args, projectRoot)` -- if error returned, return errorResponse immediately
    3. Get graph: `const cached = await getGraph(projectRoot, changedFiles)` -- pass changed files for staleness check
    4. **Per-file risk scoring (REVIEW-01):**
       - For each changed file, find node via `graph.filterNodes` matching filePath + kind="file"
       - If found: get centrality from `cached.centralities`, classify risk, compute blast radius count via `blastRadius(graph, nodeId, 4).length`
       - If not found: LOW risk, 0 centrality, 0 blast_radius_count
       - Build summary: count high/medium/low
    5. **Dependency change detection (REVIEW-02 / D-09):**
       - Open database: `openDatabase(getGraphDbPath(projectRoot))`
       - Query edges involving changed files via `getEdgesForFiles()`
       - Report all edges where source OR target is a changed file as `new_edges` (edges involving changed files -- this is the pragmatic approach from RESEARCH.md Open Question 2: report current edges rather than attempting before/after diff)
       - Note: Label them as "edges involving changed files" rather than strictly "new" since we lack a before-snapshot. The data is still useful -- it shows what dependency relationships the changed files participate in.
    6. **Circular dependency detection (REVIEW-02 / D-10):**
       - Collect node IDs for changed files
       - Run `detectCycles(graph, changedNodeIds)`
       - Convert node IDs in cycle paths to file paths for readability
    7. **Convention compliance (REVIEW-03):**
       - Read conventions.md from `getCodescopePath(projectRoot)/conventions.md`
       - If exists: parse with `parseConventions()`, match conventions whose `files` array includes any changed file
       - For each match: include convention name, adoption_pct, confidence, evidence
       - If conventions.md doesn't exist: empty array
    8. **Cross-community analysis (D-08):**
       - Query communities for changed file node IDs via `getFileCommunities()`
       - Group files by community_id
       - Flag if `communities_touched >= 3`
       - Build `community_breakdown` array with community_id, label, files list
    9. Close database
    10. Build `ReviewData` response per D-01 shape and return `okResponse(data, buildMetadata(projectRoot, startMs))`
