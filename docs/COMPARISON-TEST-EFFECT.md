# CodeScope Comparison Test — Effect-TS

## Why Effect-TS

[Effect](https://github.com/Effect-TS/effect) is the most convention-dense TypeScript project in the ecosystem. It's essentially Haskell/ZIO ported to TypeScript — a completely different programming model using algebraic effects, typed errors, dependency injection via the type system, and generator-based control flow.

Unlike h3 (our first comparison target with 14 criteria), Effect has **19 scorable conventions** that diverge so far from standard TypeScript that vanilla Claude will produce code that looks like a *different language* than the actual codebase.

| Convention | What Effect does | What Claude will likely do |
|------------|-----------------|---------------------------|
| **Async/control flow** | `Effect.gen(function*() { yield* ... })` | `async/await` with Promises |
| **Generator delegation** | `yield*` (delegating yield) — always | `yield` (compiles but wrong behavior) |
| **Function API shape** | `dual(2, impl)` for data-first AND data-last | Single calling convention |
| **Error handling** | `Data.TaggedError("Name")<{ fields }>` | `class MyError extends Error` |
| **Dependency injection** | `Context.Tag("Svc")<Self, Shape>()` | Constructor injection or plain interface |
| **Import style** | `import * as X from "./Y.js"` (namespace + .js) | `import { X } from "./Y"` |
| **Internal imports** | `"../Effect.js"` (subpath, no barrel) | `import { Effect } from "effect"` |
| **Module structure** | Public facade re-exports from `internal/` impl | Implement directly in the public module |
| **Public file naming** | PascalCase: `Stream.ts`, `Effect.ts` | camelCase: `stream.ts` |
| **Internal file naming** | camelCase: `core-stream.ts`, `stream.ts` | Same as public (no distinction) |
| **Barrel files** | Auto-generated by `pnpm codegen` — DO NOT EDIT | Manually add exports to `index.ts` |
| **Style: semicolons** | No semicolons | Semicolons everywhere |
| **Style: array syntax** | `Array<string>` (generic) | `string[]` (shorthand) |
| **Style: quotes** | Double quotes `"..."` | Single quotes `'...'` |
| **`import type`** | Always separate type imports | Mix type and value imports |
| **JSDoc format** | `@since 2.0.0`, `@category mapping`, bold `**Details**` / `**Example**` headers | Standard `/** */` or no JSDoc |
| **Testing framework** | `it.effect()` from `@effect/vitest` | Standard `it()` from vitest |
| **Test assertions** | `assert.strictEqual` / `assert.deepStrictEqual` | `expect().toBe()` / `expect().toEqual()` |
| **Changeset** | `.changeset/` file required for every PR | No changeset |

These patterns are **internally consistent but externally alien**. You have to read the source to know them — Claude's training data on standard TypeScript actively works against it here.

## The Task

> **"Add a `Stream.slidingWindow` combinator to the Effect library that collects stream elements into overlapping windows of a specified size, emitting each window as a `Chunk`."**

This task forces Claude to hit the most conventions simultaneously:

1. **Public API** — must add signature to `packages/effect/src/Stream.ts` (PascalCase, facade only)
2. **Internal implementation** — must implement in `packages/effect/src/internal/core-stream.ts` (camelCase, real logic)
3. **`dual()` wrapper** — function must support both `pipe(stream, Stream.slidingWindow(3))` and `Stream.slidingWindow(stream, 3)`
4. **`Effect.gen` + `yield*`** — implementation should use generator pattern, not async/await
5. **`.js` imports** — all imports must use `.js` extension
6. **No barrel imports** — must import `"../Chunk.js"` not `"effect"`
7. **Namespace imports** — `import * as Chunk from "../Chunk.js"`, not `import { Chunk }`
8. **JSDoc** — must include `@since 3.x.0`, `@category combinators`, bold headers
9. **Tests** — must use `it.effect()` + `assert.deepStrictEqual` from `@effect/vitest`
10. **No semicolons**, double quotes, `Array<T>` syntax
11. **Changeset** — must create `.changeset/` entry

### Exact Prompt (Use This Verbatim in Both Sessions)

```
Add a Stream.slidingWindow combinator to the Effect library.

Requirements:
- Takes a stream and a window size N, emits overlapping windows of N consecutive elements
- Each window is emitted as a Chunk (e.g., for stream [1,2,3,4,5] with window size 3: [1,2,3], [2,3,4], [3,4,5])
- Should support both piped usage (pipe(stream, Stream.slidingWindow(3))) and direct call (Stream.slidingWindow(stream, 3))
- Only start emitting once the first full window is collected
- Handle edge cases: window size larger than stream, window size of 1, empty stream
- Add the public API signature with proper documentation
- Add the internal implementation
- Write tests covering: basic sliding window, window size 1, window larger than stream, empty stream, and a stream of strings
```

## Expected Results

### Vanilla Claude Will Almost Certainly:

- **Write `async/await`** instead of `Effect.gen(function*() { yield* ... })` — this is the single biggest tell
- **Create a class** (`class SlidingWindow`) or use closures — not the `dual()` pattern
- **Put everything in one file** — miss the public facade + internal implementation split
- **Use `import { ... } from "..."` without `.js`** — standard TS import style
- **Import from `"effect"` barrel** — forbidden in internal source code
- **Write `expect().toBe()`** — standard vitest, not `@effect/vitest`
- **Use semicolons, single quotes, `T[]`** — standard TS defaults
- **Skip the changeset** — not a common convention outside specific projects
- **Write standard JSDoc** — no `@since`, `@category`, or bold section headers
- **Use `Array<A>` for windows** — not `Chunk<A>`

**Prediction: Vanilla scores 0-3/19.** The conventions are so alien that Claude's standard TS training actively produces wrong code for nearly every criterion.

### CodeScope Should:

- **Detect the `dual()` pattern** from AST analysis of existing Stream combinators
- **Detect `Effect.gen` + `yield*`** as the dominant control flow pattern
- **Detect `.js` import extensions** and namespace import style
- **Detect the public/internal split** from module structure analysis
- **Detect `@effect/vitest`** as the test framework with `it.effect()` pattern
- **Detect style rules** (no semicolons, double quotes, `Array<T>`)
- **Identify `Chunk`** as the standard collection type from usage patterns

**Prediction: CodeScope scores 13-19/19.**
